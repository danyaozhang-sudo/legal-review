# 合同审阅分析工作流 v1.1

**适用范围：** 法律审阅技能的标准操作流程，适用于投资方/总包方对建设工程类合同的分析。

**新增支持：** 起草+审查联合模式（v1.1），可在审阅已有合同的同时，从零起草新合同或补充条款。

**前置条件：** 已完成 `contract-types.md`（合同类型定义）、`legal-provisions.md`（合法条款库）、`risk-checklist.md`（风险清单）三项准备工作。

**输出产物：** 对话内结构化摘要 + （可选）.docx 正式报告

---

## Step 1: 接收输入与任务类型判断

### 确认文件格式
- 识别文件扩展名：`.docx` / `.pdf` / `.txt`
- 如果是图片类PDF（扫描件），标注需要OCR处理
- 记录文件名称和大小

### 确认我方立场
- **投资方**：侧重资金安全、股权保护、退出机制
- **总包方**：侧重工程款支付、工期保障、签证变更
- 确认立场后，后续分析优先保护我方核心利益

### 确认合同类型
- 对照 `contract-types.md` 判断合同类型（见 Step 3）
- 常见类型：建设工程施工合同、EPC总承包合同、设计合同、监理合同、采购合同、咨询服务合同

### 记录审阅信息
在分析笔记中记录以下内容：

| 字段 | 内容 |
|------|------|
| 审阅日期 | 20XX年XX月XX日 |
| 审阅目的 | 立项/尽调/签约前审查/履约监控 |
| 文件名称 | [文件名] |
| 我方立场 | 投资方 / 总包方 |
| 合同类型 | [类型] |
| 审阅人 | [可不填] |

## Step 2: 读取合同内容

### PDF 文件处理
使用 PyMuPDF（fitz）或 pdfminer 提取文本：

```python
import fitz  # PyMuPDF

def extract_pdf_text(path):
    doc = fitz.open(path)
    text = ""
    for page in doc:
        text += page.get_text()
    return text
```

> ⚠️ 注意：如果是扫描件PDF（图片扫描），需要先进行OCR识别，推荐使用 `pytesseract` 或在线OCR工具。

### DOCX 文件处理
使用 python-docx 读取：

```python
from docx import Document

def extract_docx_text(path):
    doc = Document(path)
    paragraphs = [p.text for p in doc.paragraphs]
    return "\n".join(paragraphs)
```

### 纯文本文件
直接读取使用。

### 关键要素提取
从合同正文中提取以下核心信息：

| 要素 | 说明 |
|------|------|
| 合同名称 | 完整标题 |
| 签约方 | 甲方（发包方）、乙方（承包方）全称 |
| 合同标的 | 工程名称、范围、内容 |
| 合同价款 | 金额（大写+小写）、计价方式 |
| 工期/期限 | 开工日期、竣工日期、总日历天数 |
| 签约日期 | 合同签订日期 |
| 签署地点 | 签约地点 |
| 生效条件 | 生效日期或条件 |
| 主要附件 | 附件名称列表 |

## Step 3: 识别合同类型

### 对照 contract-types.md
根据合同内容和特征，对照 `contract-types.md` 中的定义，判断合同所属类型。

### 主要合同类型（参考）

| 类型 | 特征关键词 |
|------|-----------|
| 建设工程施工合同 | 土建、安装、装饰、市政 |
| EPC总承包合同 | 设计-采购-施工、整体交付 |
| 设计合同 | 图纸、设计方案、深化设计 |
| 监理合同 | 旁站、巡视、平行检验 |
| 采购合同 | 材料设备供应、品牌规格 |
| 咨询服务合同 | 咨询、评估、代理、顾问 |

### 记录适用法律框架
根据合同类型，标注适用的法律法规：

- 建设工程施工合同 → 《民法典》《建设工程施工合同司法解释》
- EPC总承包合同 → 《民法典》《建设工程工程总承包计价计量规范》
- 其他类型 → 对应专项法规

## Step 4: 提取关键条款

### 条款分类整理
将合同全文按以下类别提取关键条款，逐条编号（如 C1、C2…）：

| 类别 | 条款内容 |
|------|---------|
| **标的条款** | 工程范围、内容、标准 |
| **价款条款** | 合同价、调整方式、支付节点 |
| **工期条款** | 开工、竣工、节点工期、延期责任 |
| **质量条款** | 验收标准、保修期、质量保证金 |
| **变更条款** | 变更范围、变更流程、变更计价 |
| **签证条款** | 现场签证、确认时效 |
| **支付条款** | 预付款、进度款、结算款、质保金 |
| **违约条款** | 违约金比例、赔偿范围 |
| **解除条款** | 解除条件、解除程序 |
| **争议条款** | 争议解决方式（诉讼/仲裁）、管辖约定 |
| **其他条款** | 不可抗力、保险、知识产权等 |

### 条款编号规则
- C = Contract clause
- 示例：C-01, C-02…
- 后续引用可直接使用编号

## Step 5: 合法性分析

### 对照 legal-provisions.md
将每条关键条款（C-01、C-02…）逐一对照 `legal-provisions.md` 中的法律规定进行合规性审查。

### 合法性判断标准

| 结论 | 含义 |
|------|------|
| ✅ 合法 | 条款符合法律规定，有效 |
| ⚠️ 存在风险 | 条款可能损害我方权益，需关注 |
| ❌ 可能无效 | 条款违反强制性规定，可能被认定无效 |
| ❓ 需专业判断 | 条款涉及专业判断，建议咨询律师 |

### 输出格式

**条款编号 | 条款摘要 | 合法性结论 | 相关法条 | 说明**

示例：

| C-07 | 发包人逾期付款按日万分之三支付违约金 | ⚠️ 存在风险 | 《民法典》第585条 | 违约金过分高于造成的损失，我方可主张调减 |
| C-12 | 承包人放弃工期索赔权利 | ❌ 可能无效 | 《民法典》第153条 | 排除对方主要权利的格式条款可能无效 |

## Step 6: 风险评估

### 对照 risk-checklist.md
根据 `risk-checklist.md` 中的风险清单，对每条可能存在问题的条款进行风险识别。

### 风险等级定义

| 等级 | 定义 | 影响 |
|------|------|------|
| 🔴 高风险 | 可能导致重大损失或法律后果 | 需立即修改或拒绝 |
| 🟡 中风险 | 可能造成一定损失或不利 | 建议修改 |
| 🟢 低风险 | 影响有限，必要时可接受 | 可选择性修改 |

### 风险评估要素
- 风险来源（对方违约/市场变化/政策调整）
- 可能后果（金钱损失/工期延误/法律纠纷）
- 发生概率（高/中/低）
- 我方承受能力

### 输出格式

**风险编号 | 涉及条款 | 风险描述 | 风险等级 | 可能后果**

示例：

| R-01 | C-07 | 违约金上限约定过低，无法覆盖实际损失 | 🟡 中 | 对方违约时我方实际损失无法充分赔偿 |
| R-02 | C-15 | 管辖权约定对我方极不方便 | 🔴 高 | 发生争议时需远距离应诉，成本高 |

---

## Step 7: 修改建议

### 针对每个高/中风险条款提出修改建议

### 修改建议格式

**条款编号：C-XX**
- **风险等级**：🔴 高 / 🟡 中
- **原文**：[原始条款文本]
- **建议修改为**：[修改后的条款文本]
- **修改理由**：[法律依据 + 商业合理性]
- **优先级**：必须修改 / 建议修改 / 可协商

### 示例

**条款编号：C-07（价款条款）**
- **风险等级**：🟡 中
- **原文**："发包人逾期付款按日万分之三支付违约金"
- **建议修改为**："发包人逾期付款按日万分之五支付违约金，且承包人有权暂停施工"
- **修改理由**：《民法典》第585条允许约定违约金；日万分之三明显低于融资成本，无法起到约束作用
- **优先级**：建议修改

### 修改优先级分类

| 优先级 | 含义 |
|--------|------|
| 🔴 必须修改 | 不修改则拒绝签约或无法接受 |
| 🟡 建议修改 | 争取修改，降低风险 |
| 🟢 可协商 | 如对方坚持，可接受现有文本 |

## 起草分支（联合模式专用）

### Step 1a：识别任务类型

**判断依据**：用户输入匹配以下关键词任一，即进入起草分支：
- 「起草」「写一份」「生成」「帮我起草」
- 配合合同类型词：「EPC合同」「NDA」「服务协议」「合作框架」等

**联合模式识别**：同时出现起草词 + 审阅词（如「起草并审查」「写完审一遍」）→ 完整执行起草分支 + 审阅主流程

**纯起草模式**：仅有起草词 → 仅执行 Step 2a~3a，生成初稿后询问用户是否需要法律审查

### Step 2a：调用 legal-contract-gen 收集变量

调用 `legal-contract-gen` 技能的「交互式问答流程」，按以下顺序收集：

**第一轮：通用变量**

| 变量 | 问题 | 默认值 |
|------|------|-------|
| 甲方名称 | 甲方（披露方/服务提供方）的全称？ | — |
| 乙方名称 | 乙方（接收方/客户方）的全称？ | — |
| 签署日期 | 协议签署日期？（留空则填写当日） | 签署当日 |
| 适用法律 | 本协议适用哪个司法管辖区的法律？ | 中华人民共和国法律 |
| 争议解决 | 选仲裁还是诉讼？如选仲裁，指定哪个机构？ | 诉讼 / 被告所在地法院 |
| 语言 | 文书使用中文还是英文？ | 中文 |

**第二轮：专属变量**（根据合同类型）

参照 `legal-contract-gen` SKILL.md 中对应合同类型的变量表，按批次提问，不一次性列出全部问题。

**立场传递**：确认用户立场（投资方/总包方），整份合同以该立场编写。

### Step 3a：生成合同初稿

- 调用 `legal-contract-gen` 对应模板，填入变量
- 甲方/乙方的权利义务根据立场调整（投资方：侧重资金安全+退出；总包方：侧重工期保障+变更索赔）
- 所有待填内容用 `[待填写：xxx]` 标注占位符
- 输出合同全文

**输出后判断**：
- 联合模式 → 立即进入审阅主流程 Step 4
- 纯起草模式 → 告知用户已完成初稿，询问是否需要法律审查
